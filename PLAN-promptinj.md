# plan: llm-fw implementation

## Phases

### Phase 1 — Project Scaffold
- Init `package.json` (name: `llm-fw`, type: module, bin: `llm-fw`)
- `tsconfig.json` targeting Node 22, ESM
- Folder structure:
  ```
  src/
    cli/          # setup, start, stop, status commands
    proxy/        # HTTPS proxy core (CONNECT + TLS termination)
    detection/    # normalize, heuristic, embedding, judge, pipeline
    dashboard/    # HTTP server, SSE, playground
    config/       # cosmiconfig loader + defaults
  data/
    attacks.json  # ~100 canonical injection templates
  ```
- Install deps: `node-forge`, `@huggingface/transformers`, `cosmiconfig`, `tsx` (dev)

### Phase 2 — Attack Template Library (`data/attacks.json`)
- Curate ~100 canonical prompt injection / jailbreak strings
- Sources: PromptInject dataset, JailbreakChat archive, manual adversarial patterns
- Format: flat string array (no labels needed — Stage 2 is similarity-only)
- This file is the entire "vector database" — embedded into memory at startup

### Phase 3 — Detection Pipeline (`src/detection/`)
Five modules + a pipeline orchestrator:

**3a. Input normalizer** (`normalize.ts`)
- Pure TS, zero deps
- Unicode NFKC normalization
- Lowercase, strip zero-width characters, collapse excessive punctuation
- Called before Stage 1 and before embedding in Stage 2

**3b. Heuristic scorer** (`heuristic.ts`)
- Pure TS, zero deps
- Weighted regex clusters applied to normalized input
- Returns `{ score: number, matches: string[] }`

**3c. Embedding checker** (`embedding.ts`)
- Loads `Xenova/all-MiniLM-L6-v2` via `@huggingface/transformers` (ONNX/WASM)
- On init: embed all `attacks.json` strings -> `Float32Array[]` cached in memory
- Per-request:
  1. Normalize input
  2. If input > 300 tokens: sliding-window chunk (200-token chunks, 50-token overlap)
  3. Embed each chunk individually
  4. Score = max cosine similarity across all chunks
- Returns `{ similarity: number, nearest: string, chunkCount: number }`
- Risk: model download (~30MB) on first use — handle in `setup` command

**3d. Judge LLM client** (`judge.ts`)
- HTTP call to local Ollama API (`/api/generate`)
- Ollama params: `{ "num_predict": 1, "temperature": 0 }`
- Parses single-token response: SAFE | MALICIOUS
- Skipped entirely if `judgeEnabled: false`

**3e. Payload parsers** (`parsers.ts`)
- Interface: `{ supports(path): boolean; extractPrompts(body): string[] }`
- Anthropic parser: extracts `messages[].content` (role: user) + optional `system`
- Gemini parser: extracts `contents[].parts[].text` + `systemInstruction`
- Fallback: returns empty array for unknown paths / non-JSON bodies (pass through)

**3f. Pipeline orchestrator** (`pipeline.ts`)
- Selects parser by request path
- Extracts prompts; if none extracted, returns PASS
- For each prompt: normalize -> Stage 1 -> (Stage 2 if score 20-49) -> (Stage 3 async/sync)
- Short-circuits on first BLOCK
- Returns `PipelineResult: { action: 'block'|'pass'|'warn', stage, score, similarity, verdict }`

### Phase 4 — Proxy Core (`src/proxy/`)
This is the hardest phase.

**4a. CA + cert factory** (`certs.ts`, uses `node-forge`)
- `generateCA()` -> self-signed root CA stored at `~/.llm-fw/ca.{crt,key}`
  - Set `~/.llm-fw/` permissions to `0700` (macOS/Linux) or user-only ACL (Windows)
- `generateHostCert(hostname)` -> cert signed by CA with `subjectAltName` = hostname
  - Cache per hostname in memory
- Required SAN extension:
  ```javascript
  extensions: [{ name: 'subjectAltName', altNames: [{ type: 2, value: hostname }] }]
  ```

**4b. Proxy server** (`proxy.ts`)
- Dual-mode listener:
  - **Mode A (primary):** port 8080, handles HTTP CONNECT requests (HTTPS_PROXY)
  - **Mode B (sinkhole, opt-in):** port 443, handles direct TLS connections
    - Drop privileges via `process.setuid()` after binding (Linux/macOS)
- On CONNECT (Mode A) / incoming TLS (Mode B):
  1. Present per-host cert (generated by `certs.ts`)
  2. Decrypt and buffer request body
  3. Run pipeline (via `pipeline.ts`)
  4. PASS: pipe response socket directly (zero-copy streaming, no buffering)
  5. BLOCK: return HTTP 403 JSON + emit block event to dashboard
- Response streaming: once a request passes, pipe upstream response socket directly
  to downstream socket without buffering — preserves SSE / chunked streaming

**4c. Upstream resolver** (`upstream.ts`)
- Uses `node:dns/promises Resolver` configured with external servers:
  ```typescript
  const resolver = new Resolver();
  resolver.setServers(['1.1.1.1', '8.8.8.8']);
  ```
- Bypasses local hosts file entirely — prevents DNS loop when Mode B is active
- Resolves per-request with TTL-respecting LRU cache (avoids CDN IP staleness)

### Phase 5 — CLI (`src/cli/`)

**setup** (base: no admin required for Mode A)
1. Generate CA cert (`certs.ts`) + set `~/.llm-fw/` to `0700`/user-ACL
2. Install CA to OS trust store:
   - Windows: `certutil -addstore Root <ca.crt>`
   - macOS: `security add-trusted-cert`
   - Linux: copy to `/usr/local/share/ca-certificates/`, run `update-ca-certificates`
3. Pre-download embedding model (triggers Transformers.js cache to `~/.llm-fw/models/`)
4. `--sinkhole` flag: additionally write hosts file entries (requires elevation prompt)

**start**
1. Load config
2. Init embedding model (warm cache from setup)
3. Start dashboard server
4. Start proxy server (mode determined by config)
5. Register exit lifecycle hooks (before server start):
   ```typescript
   const cleanup = () => { if (sinkholeActive) restoreHostsFile(); };
   process.on('SIGINT', cleanup);
   process.on('SIGTERM', cleanup);
   process.on('uncaughtException', (err) => { cleanup(); process.exit(1); });
   ```
6. Write PID file to `~/.llm-fw/llm-fw.pid`

**stop**
1. Read PID file, send SIGTERM (triggers cleanup hook in the running process)
2. Wait for clean exit; fallback SIGKILL after 5s

**status**
- Check PID alive, print active mode, proxy port, dashboard URL, model loaded state

### Phase 6 — Dashboard (`src/dashboard/`)
- Single `node:http` server on port 7731
- In-memory ring buffer of last 100 block/warn events
- **Routes:**
  - `GET /` -> serve embedded HTML template (no bundler; template string in source)
  - `GET /events` -> SSE: replay stored events on connect, then push live events
  - `GET /api/events` -> JSON array, `?page=N&limit=50`
  - `POST /api/test` -> run full pipeline on submitted `{ prompt: string }` body;
    return `{ stage1, stage2, stage3, verdict }` breakdown for Playground tab

- **Events tab** (HTML): table with columns: time, stage, score, similarity, target, payload preview
  - Auto-updates via `EventSource` JS API (no framework)
  - Row colors: heuristic=yellow, embedding=orange, judge=red

- **Playground tab** (HTML):
  - Textarea: prompt input
  - "Analyze" button -> POST /api/test
  - Results panel showing per-stage breakdown:
    - Stage 1: matched phrases list + total score
    - Stage 2: nearest template + cosine similarity (color-coded bar)
    - Stage 3: Ollama verdict (greyed out if disabled)
    - Final verdict badge: PASS / WARN / BLOCK

### Phase 7 — Config System (`src/config/`)
- `cosmiconfig('llm-fw')` — resolves `.llm-fw.json` / `llm-fw.config.js` / package.json field
- Deep merge with hardcoded defaults
- Env var overrides: `LLM_FW_PROXY_PORT`, `LLM_FW_PROXY_MODE`, `LLM_FW_EMBEDDING_BLOCK_THRESHOLD`, etc.
- Validate with simple runtime checks (no zod — keep deps minimal)

### Phase 8 — Integration & Hardening
- End-to-end smoke test: start proxy in Mode A, set HTTPS_PROXY, send known injection, assert 403 + dashboard event
- Verify sliding-window: send long input with injection buried at end, assert detected
- Verify SAN: connect with strict TLS validation, confirm no cert error
- Benchmark embedding latency (target < 50ms cold, < 20ms warm)
- Platform test on Windows 11 (primary target)
- Verify hosts restore: kill -9 the process, confirm hosts file is clean on next `setup` run
  (Note: SIGKILL cannot be caught — document that Mode B users should prefer `llm-fw stop`)
- Edge cases:
  - Non-JSON request bodies: pass through untouched
  - `stream: true` responses: confirm zero-copy pipe works with chunked encoding
  - Elevation failure: clear error message with fix instructions

                                          \-> 6 (dashboard) -/
                7 (config) feeds all

Phase 4 depends on Phase 3 (needs pipeline) and Phase 7 (needs config).
Phase 5 depends on Phase 4.
Phase 8 depends on everything.

## Key Risks

| Risk | Mitigation |
|------|-----------|
| Windows admin for trust store | Required once for CA install; Mode A (HTTPS_PROXY) avoids hosts edit entirely |
| DNS loop in Mode B (sinkhole) | External DNS resolver (1.1.1.1) bypasses local hosts file at the socket level |
| CDN IP rotation | TTL-respecting LRU cache on the external resolver; no stale pre-resolved IPs |
| Transformers.js WASM slow on first inference | Warm on startup during `setup`; show progress; store in `~/.llm-fw/models/` |
| TLS SAN rejection by strict clients | `node-forge` adds `subjectAltName` to every generated cert |
| CA private key exposure | `~/.llm-fw/` chmod 0700 / Windows user-only ACL |
| SIGKILL leaves hosts modified (Mode B) | Document; `setup` detects and cleans stale entries on next run |
| Prompt dilution on long inputs | Sliding-window chunking in Stage 2 with max-score aggregation |
| Stage 3 latency budget | Async default (judgeBlock: false) adds 0ms to p95; num_predict: 1 minimizes generation |

### Phase 9 — Test Suite (`test/`)

Test runner: **Vitest**. Test files mirror `src/` structure: `test/detection/`, `test/proxy/`, `test/dashboard/`.

**9a. Unit tests**
- `normalize.test.ts` — Unicode edge cases, zero-width chars, homoglyphs
- `heuristic.test.ts` — all weight categories, threshold boundaries (0/20/49/50), regex edge cases
- `embedding.test.ts` — cosine similarity math, chunk split boundaries and overlap, max-score aggregation (mock model)
- `parsers.test.ts` — Anthropic multi-turn, Gemini, non-JSON fallback, missing fields
- `pipeline.test.ts` — stage routing at score 0/25/60, BLOCK short-circuit, empty-prompt pass-through

**9b. Integration tests**
- `pipeline.integration.test.ts` — full pipeline with real embedding model; known injections -> BLOCK, clean messages -> PASS
- `proxy.integration.test.ts` — in-process proxy, mock upstream, CONNECT flow, assert 200/403
- `dashboard.integration.test.ts` — start dashboard server, POST /api/test, GET /api/events pagination

**9c. E2E tests (Mode A only — no hosts mutation)**
- Start proxy with HTTPS_PROXY, send Anthropic-shaped injection -> assert 403 + SSE event
- Clean prompt -> assert forwarded to mock upstream
- Playground POST /api/test -> assert stage breakdown structure

**Coverage gate:** 90% line coverage on `src/detection/` enforced in CI. E2E runs on CI only (not pre-commit).
